# x CTO-to-Code Loop Principles

- `x` is first a CTO-to-code execution loop, not a company operating system.
- Root supplies the engineering directive and keeps final merge authority.
- The active roles are `cto`, `engineer`, and `reviewer`; the main agent orchestrates and writes state.
- All roles load repository project context before package-specific instructions.
- CTO co-creation is the entry. The Engineering Loop is the lower execution layer after an accepted CTO Intake Brief.
- CEO/Product/Growth layers may later produce root-ready CTO inputs, but they do not participate in this MVP.
- Every run must move through repo intake, codebase investigation, accepted CTO Intake Brief, Technical Contract, Engineer Task, implementation evidence, review, fix loop if needed, and merge-back recommendation.
- Do not create a Technical Contract without an accepted CTO Intake Brief.
- Do not start implementation without a Technical Contract and Engineer Task.
- Do not start review without patch evidence and verification evidence.
- Do not close accepted while required verification is missing or review has blocking findings.
- Merge-back means recommendation only; do not merge, push, or open PR unless root explicitly asks.
